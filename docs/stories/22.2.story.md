# Story 22.2: Dispatch Engine with multiclaude CLI Wrapper

**Status:** draft
**Epic:** 22 — Self-Driving Development Pipeline
**Depends on:** Story 22.1
**Blocks:** Stories 22.4, 22.5, 22.7, 22.8

## User Story

As a developer,
I want a dispatch engine that wraps the multiclaude CLI,
So that ThreeDoors can create workers, list workers, get history, and remove workers programmatically.

## Acceptance Criteria

**Given** the need to interact with multiclaude from within ThreeDoors
**When** the dispatch engine is implemented
**Then:**

- AC1: `Dispatcher` interface defined in `internal/dispatch/dispatcher.go` with methods: `CreateWorker(ctx context.Context, task string) (workerName string, err error)`, `ListWorkers(ctx context.Context) ([]WorkerInfo, error)`, `GetHistory(ctx context.Context, limit int) ([]HistoryEntry, error)`, `RemoveWorker(ctx context.Context, name string) error`
- AC2: `CLIDispatcher` concrete type implements `Dispatcher` — wraps `os/exec` calls to `multiclaude` binary
- AC3: `CommandRunner` interface (`Run(ctx context.Context, name string, args ...string) ([]byte, error)`) injected into `CLIDispatcher` for testability
- AC4: `BuildTaskDescription(item QueueItem) string` method constructs rich worker prompt from task text, context, acceptance criteria, scope constraints, and standard suffix (commit signing, fork workflow, no co-authored-by)
- AC5: `CheckAvailable(ctx context.Context) error` validates `multiclaude` is on PATH via `exec.LookPath`
- AC6: `WorkerInfo` struct with fields: Name, Status, Branch, Task
- AC7: `HistoryEntry` struct with fields: WorkerName, Status, PRNumber, PRURL, Summary, CompletedAt
- AC8: Unit tests with mock `CommandRunner` for all four dispatch operations plus availability check
- AC9: Error wrapping follows `fmt.Errorf("dispatch %s: %w", op, err)` pattern

## Technical Notes

- Accept interfaces, return concrete types: `NewCLIDispatcher(runner CommandRunner) *CLIDispatcher`
- `context.Context` is always the first parameter per project conventions
- The standard suffix appended to every worker task description:
  ```
  IMPORTANT: Sign all commits (git commit -S). Do NOT add Co-Authored-By for AI.
  This is a fork workflow — PR targets upstream arcaven/ThreeDoors via
  gh pr create --repo arcaven/ThreeDoors --head arcavenai:<branch>.
  ```
- Parse `multiclaude worker list` output to extract WorkerInfo fields
- Parse `multiclaude repo history` output to extract HistoryEntry fields
- Use `exec.CommandContext` for timeout support on all subprocess calls
- Default timeout: 30 seconds per CLI call
