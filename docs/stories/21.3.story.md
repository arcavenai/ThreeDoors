# Story 21.3: Canonical ID Mapping (SourceRef)

**Status:** Done (partial — PR #132: SourceRef type and Task methods; TaskPool integration AC4-AC6 deferred)
**Epic:** 21 — Sync Protocol Hardening

## User Story

As a ThreeDoors user with tasks in multiple providers,
I want tasks to be permanently linked across providers,
So that deduplication is reliable and write-back works to all sources.

## Acceptance Criteria

**Given** a task exists in multiple providers
**When** the task is loaded from each provider
**Then:**

- AC1: `SourceRef` struct in `internal/core/task.go` with Provider (string) and NativeID (string) fields
- AC2: `Task` gains `SourceRefs []SourceRef` field with `yaml:"source_refs,omitempty" json:"source_refs,omitempty"`
- AC3: Backward compatibility: if `SourceRefs` is empty, fall back to `SourceProvider` field
- AC4: `FindBySourceRef(provider, nativeID string) *Task` lookup on TaskPool
- AC5: Identity resolution: on LoadTasks, match by SourceRef first, then heuristic dedup, then create new
- AC6: Write routing: `SaveTask` uses SourceRefs to propagate changes to all providers that know a task
- AC7: Schema version in config bumped to 2; migration function converts `SourceProvider` → `SourceRefs[0]`
- AC8: Unit tests for SourceRef matching, migration, backward compatibility, write routing

## Technical Notes

- SourceRef replaces the single `SourceProvider` field for multi-provider identity
- Migration: on first load, if SourceRefs is nil/empty but SourceProvider is set, create `SourceRefs = []SourceRef{{Provider: SourceProvider, NativeID: ID}}`
- Heuristic dedup: normalized text match within 24h creation window (existing DuplicateDetector logic)
- When dedup links two tasks, the non-canonical one gains a SourceRef pointing to the canonical one's provider
- Write routing: iterate SourceRefs, call SaveTask on each provider's active instance via Registry
- Make the zero value useful: empty SourceRefs behaves identically to current behavior
