# Story 19.2: Jira Read-Only Provider

**Status:** draft
**Epic:** 19 ‚Äî Jira Integration
**Blocks:** Stories 19.3, 19.4

## User Story

As a ThreeDoors user with Jira,
I want my Jira issues to appear as tasks in ThreeDoors,
So that I can use the Three Doors selection for my sprint work.

## Acceptance Criteria

**Given** a configured Jira connection with valid JQL
**When** LoadTasks is called
**Then:**

- AC1: `JiraProvider` in `internal/adapters/jira/jira_provider.go` implementing `core.TaskProvider`
- AC2: `LoadTasks()` executes configured JQL, paginates all results, maps to `[]*core.Task`
- AC3: Field mapping: issue key ‚Üí ID, summary ‚Üí Text, statusCategory ‚Üí Status, priority ‚Üí Effort, project+labels ‚Üí Context
- AC4: `SaveTask/SaveTasks/DeleteTask/MarkComplete` return `core.ErrReadOnly`
- AC5: `Watch()` returns `nil`; `HealthCheck()` tests API connectivity
- AC6: `Name()` returns `"jira"`
- AC7: Adapter factory function: `Factory(config *core.ProviderConfig) (core.TaskProvider, error)`
- AC8: Contract tests pass via `adapters.RunContractTests`
- AC9: Table-driven field mapping tests for all status/priority combinations

## Technical Notes

- Status mapping defaults: `statusCategory` key ‚Üí `new`=todo, `indeterminate`=in-progress, `done`=complete
- Effort mapping defaults: Highest/High=deep-work, Medium=medium, Low/Lowest=quick-win
- Context: `fmt.Sprintf("[%s] %s", project.Key, strings.Join(labels, ", "))`
- Task.ID = Jira issue key (e.g., `PROJ-42`) ‚Äî human-readable, unique
- Task.SourceProvider = `"jira"`
- Use `fmt.Fprintf` not `WriteString(Sprintf())` for string building
- Accept interfaces, return concrete types
