# Story 23.1: Cobra Scaffolding, Root Command, and Output Formatter

**Status:** Done (PR #170)
**Epic:** 23 — CLI Interface
**Blocks:** Stories 23.2, 23.3, 23.4, 23.5

## User Story

As a developer,
I want a Cobra-based CLI scaffold with a root command, `--json` persistent flag, and a shared output formatter,
So that all subsequent CLI commands have a consistent foundation for routing, output formatting, and error handling.

## Acceptance Criteria

**Given** the need for a CLI interface alongside the existing TUI
**When** the scaffolding is implemented
**Then:**

- AC1: `internal/cli/root.go` defines the Cobra root command with `Use: "threedoors"`, long description, and `--json` persistent flag
- AC2: `internal/cli/output.go` defines an `OutputFormatter` that supports both human-readable (tabwriter) and JSON output modes
- AC3: JSON output uses a consistent envelope: `{"schema_version": 1, "command": "<cmd>", "data": ..., "metadata": {...}}`
- AC4: JSON error responses use envelope: `{"schema_version": 1, "command": "<cmd>", "error": {"code": N, "message": "...", "detail": "..."}}`
- AC5: `cmd/threedoors/main.go` updated with subcommand detection — no args launches TUI (backward compatible), any known subcommand routes to Cobra CLI
- AC6: `go.mod` updated with `github.com/spf13/cobra` dependency
- AC7: Exit code constants defined: 0 (success), 1 (general error), 2 (not found), 3 (validation error), 4 (provider error), 5 (ambiguous input)
- AC8: `threedoors --help` shows command groups and global flags
- AC9: Unit tests for output formatter covering JSON envelope structure, human-readable table output, and error formatting

## Technical Notes

- The subcommand detection in `main.go` should check `os.Args[1]` against known subcommands before deciding TUI vs CLI routing
- Cobra's `PersistentPreRunE` is the right place to initialize the provider and set up shared state
- The output formatter should accept `io.Writer` for testability (not hardcoded to `os.Stdout`)
- Human-readable output uses `text/tabwriter` from stdlib — no additional dependencies
- Reference: `docs/research/cli-interface-design.md` sections 2, 3, 5, 7
