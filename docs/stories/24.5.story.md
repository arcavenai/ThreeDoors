# Story 24.5: TUI Proposal Review View

**Status:** draft
**Epic:** 24 — MCP/LLM Integration Server
**Depends on:** Story 24.4

## User Story

As a user,
I want to review, approve, and reject LLM-generated proposals from within the TUI,
So that I maintain full control over what changes are applied to my tasks.

## Acceptance Criteria

**Given** pending proposals exist in the proposal store
**When** the user interacts with the TUI
**Then:**

- AC1: Badge indicator on doors view showing pending proposal count: `[3 suggestions]`
- AC2: Proposal review view accessible via `S` key from doors view or `:suggestions` command
- AC3: Split pane layout: proposals listed on left, affected task detail on right
- AC4: Quick actions: Enter = approve, Backspace = reject, Tab = skip, Ctrl+A = approve all visible
- AC5: On approve: proposal status set to `approved`, payload applied to task via `TaskProvider.SaveTask()`, enrichment DB updated
- AC6: On reject: proposal status set to `rejected` with timestamp
- AC7: Stale proposals visually distinguished (dimmed or strikethrough) with tooltip "Task changed since this suggestion"
- AC8: Preview mode: before approving, show diff of what the task will look like after applying the proposal
- AC9: Batch grouping: proposals for the same task grouped together for efficient review
- AC10: `j`/`k` or arrow keys navigate proposals, ESC returns to doors view
- AC11: Empty state message when no proposals pending: "No suggestions — LLM clients can propose enrichments via MCP"
- AC12: Unit tests for proposal application logic, stale detection rendering, and key bindings

## Technical Notes

- Follows Bubbletea MVU pattern — all output through `View()`, no `fmt.Println`
- Use Lipgloss for styling — badge, split pane, diff highlighting
- Approval flow should feel lightweight, not bureaucratic — quick keystrokes, not dialogs
- Applying a proposal is the only code path that translates LLM suggestions into actual task modifications
- Stale detection: compare `Proposal.BaseVersion` to `Task.UpdatedAt` at render time
- Preview mode builds a copy of the task with the proposal applied, renders both side by side
