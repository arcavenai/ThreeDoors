# Story 19.3: Jira Bidirectional Sync

**Status:** draft
**Epic:** 19 — Jira Integration

## User Story

As a ThreeDoors user,
I want completing a Jira task in ThreeDoors to transition the issue to Done in Jira,
So that my task status stays synchronized.

## Acceptance Criteria

**Given** a Jira task is marked complete in ThreeDoors
**When** the MarkComplete operation executes
**Then:**

- AC1: `MarkComplete(taskID)` calls GET transitions → finds Done transition (statusCategory.key == "done") → POST transition
- AC2: Handle 409 Conflict (concurrent transition) with retry and backoff
- AC3: WAL wrapping: `core.NewWALProvider(jiraProvider)` queues failed writes for replay
- AC4: FallbackProvider wrapping for graceful degradation when Jira is unreachable
- AC5: Local cache file (`~/.threedoors/jira-cache.yaml`) updated on successful LoadTasks, used as fallback when offline
- AC6: WAL replay checks current status before transitioning (skip if already Done)
- AC7: Unit tests for transition discovery, execution, conflict handling, WAL replay, and cache fallback

## Technical Notes

- Transitions API: `GET /rest/api/3/issue/{key}/transitions` returns available transitions
- Find transition where `to.statusCategory.key == "done"` — don't hardcode transition IDs
- `POST /rest/api/3/issue/{key}/transitions` with `{"transition": {"id": "31"}}` — returns 204 on success
- 409 means concurrent transition; retry with exponential backoff (max 3 retries)
- Cache file format: YAML list of `*core.Task` with `last_updated` timestamp
- WAL replay: before transitioning, GET current status; if already Done, skip (idempotent)
- Atomic writes for cache file: write to `.tmp`, sync, rename
