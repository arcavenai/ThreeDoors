# Story 24.1: MCP Server Binary & Transport Layer

**Status:** Done
**Epic:** 24 ‚Äî MCP/LLM Integration Server
**Blocks:** Stories 24.2, 24.3

## User Story

As a developer,
I want a standalone MCP server binary that implements the MCP protocol over stdio and SSE transports,
So that LLM clients (Claude Desktop, Cursor, etc.) can connect to ThreeDoors and discover available capabilities.

## Acceptance Criteria

**Given** the need for a separate MCP server binary sharing `internal/` packages
**When** the server binary is created
**Then:**

- AC1: `cmd/threedoors-mcp/main.go` entry point created with `MCPServer` struct wrapping existing core components (`Registry`, `MultiSourceAggregator`, `TaskPool`, `SessionTracker`, `EnrichmentStore`)
- AC2: Server implements MCP JSON-RPC protocol with `initialize`, `initialized`, `resources/list`, `tools/list`, `prompts/list` handlers
- AC3: stdio transport mode (default) ‚Äî reads JSON-RPC from stdin, writes to stdout, suitable for Claude Desktop integration
- AC4: SSE transport mode (`--transport sse --port 8080`) ‚Äî HTTP server with Server-Sent Events for remote LLM access
- AC5: Server advertises capabilities: `resources: {subscribe: true, listChanged: true}`, `tools: {}`, `prompts: {listChanged: true}`
- AC6: `MCPMiddleware` type defined as `func(Handler) Handler` following the decorator pattern used by `WALProvider`
- AC7: Server loads ThreeDoors config from `~/.threedoors/config.yaml` and initializes core components (registry, aggregator, pool)
- AC8: `Makefile` updated with `build-mcp` target for the MCP server binary
- AC9: Unit tests for protocol handshake, capability advertisement, and transport selection

## Technical Notes

- The MCP server is a separate binary from the TUI ‚Äî they share `internal/` packages but are independently deployable
- No new storage layer ‚Äî MCP server reads from the same YAML files, JSONL logs, and SQLite enrichment DB as the TUI
- Use an existing Go MCP SDK (e.g., `github.com/mark3labs/mcp-go`) or implement minimal JSON-RPC handler
- `MCPServer` struct fields mirror the research doc: `registry`, `aggregator`, `pool`, `session`, `enrichDB`, `middleware`
- Transport selection via CLI flag, not config file ‚Äî server mode is an operational concern
