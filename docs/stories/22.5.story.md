# Story 22.5: Worker Status Polling and Task Update Loop

**Status:** in-progress
**Epic:** 22 — Self-Driving Development Pipeline
**Depends on:** Story 22.2
**Blocks:** Story 22.6

## User Story

As a user,
I want ThreeDoors to automatically check on worker status and update tasks with PR results,
So that I can see development progress without leaving the TUI.

## Acceptance Criteria

**Given** one or more queue items are in `dispatched` status
**When** the 30-second tick fires
**Then:**

- AC1: `tea.Tick` command fires every 30 seconds while any queue items are in `dispatched` status
- AC2: Tick calls `Dispatcher.GetHistory(ctx, 10)` to retrieve recent worker completions
- AC3: Worker name from history matched to queue item's `WorkerName` field; queue item status updated accordingly
- AC4: PR number and URL extracted from history and set on both the queue item (`PRNumber`, `PRURL`) and `Task.DevDispatch` (`PRNumber`, `PRStatus`)
- AC5: Task badge in doors view updates to show PR status: `[PR #N]` for open PRs, `[MERGED]` for merged PRs
- AC6: Polling stops (no more ticks scheduled) when no items are in `dispatched` status — avoids unnecessary subprocess calls
- AC7: Parse errors and subprocess failures logged via `log.Printf` but do not crash the TUI — Bubbletea `Update()` must never panic
- AC8: `WorkerStatusMsg` Bubbletea message type carries polling results through the Update chain

## Technical Notes

- Polling pattern follows existing `tea.Tick` usage in the TUI (e.g., sync status polling)
- The tick is started when the first item transitions to `dispatched` and stopped when the last dispatched item completes/fails
- `WorkerStatusMsg` is defined in `internal/tui/messages.go` or the dispatch package
- Tick interval (30s) is a constant — could be configurable in future but hardcoded for MVP
- `Dispatcher.GetHistory` is called in a `tea.Cmd` goroutine — never block the Update loop
- Match by `WorkerName` — the name is set when `CreateWorker` returns successfully
- Status mapping: multiclaude `running` → queue `dispatched`, `no-pr` → queue `failed`, `open` → queue `completed`, `merged` → queue `completed`
