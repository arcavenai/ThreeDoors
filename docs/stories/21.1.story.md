# Story 21.1: Sync Scheduler with Per-Provider Loops

**Status:** draft
**Epic:** 21 — Sync Protocol Hardening
**Blocks:** Story 21.4

## User Story

As a ThreeDoors user with multiple task sources,
I want background sync to run automatically per provider,
So that I don't have to interact with the app to discover remote changes.

## Acceptance Criteria

**Given** multiple providers are configured
**When** the sync scheduler starts
**Then:**

- AC1: `SyncScheduler` in `internal/core/sync_scheduler.go` managing per-provider `ProviderLoop` goroutines
- AC2: Each loop runs independently with configurable poll interval
- AC3: Hybrid trigger: `Watch()` channel as primary, polling as fallback (running concurrently)
- AC4: `AdaptiveInterval`: reset to min on success, multiply by 2.0 on failure (up to max), ±20% jitter
- AC5: Results fan-in to a single `chan SyncResult` for TUI consumption
- AC6: `Start(ctx context.Context)` and `Stop()` lifecycle methods
- AC7: Unit tests with fake clock, mock providers, deterministic scheduling
- AC8: No goroutine leaks on Stop() (verified via runtime.NumGoroutine or leak detector)

## Technical Notes

- Default intervals: TextFile/Obsidian = fsnotify primary, 30s-5m poll fallback; API adapters = 30s-30m poll
- Jitter prevents thundering herd: `interval * (1 + jitter * (rand.Float64()*2 - 1))`
- SyncResult type: `{Provider string, Tasks []*Task, Err error, Duration time.Duration}`
- Stop() must cancel all goroutines via context and drain the results channel
- Use `context.WithCancel` per loop for independent shutdown
- The scheduler should not directly reference SyncEngine — it calls LoadTasks() and reports results
- TUI integration: wrap results channel in a `tea.Cmd` that listens for SyncResult messages
