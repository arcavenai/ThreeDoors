# Story 21.2: Circuit Breaker per Provider

**Status:** Done
**Epic:** 21 — Sync Protocol Hardening
**Blocks:** Story 21.4

## User Story

As a ThreeDoors user,
I want a failing provider to be isolated without affecting other providers,
So that one unreachable service doesn't degrade my entire task view.

## Acceptance Criteria

**Given** a provider experiences consecutive failures
**When** the failure threshold is reached
**Then:**

- AC1: `CircuitBreaker` in `internal/core/circuit_breaker.go` with Closed/Open/Half-Open states
- AC2: Closed → Open after 5 consecutive failures within 2-minute window
- AC3: Open → Half-Open after probe interval elapses (starts 30s, doubles each cycle, max 30m)
- AC4: Half-Open → Closed on successful probe (reset failure count and interval)
- AC5: Half-Open → Open on failed probe (double probe interval)
- AC6: `Execute(fn func() error) error` method that tracks success/failure and enforces circuit state
- AC7: `State() CircuitState` method for UI integration
- AC8: Thread-safe (sync.Mutex protected)
- AC9: Table-driven state transition tests covering all paths

## Technical Notes

- CircuitState enum: `const (CircuitClosed CircuitState = iota; CircuitOpen; CircuitHalfOpen)`
- In Open state, `Execute()` returns `ErrCircuitOpen` without calling the function
- In Half-Open state, `Execute()` allows exactly one call (the probe)
- Failure window: only count failures within the last 2 minutes; older failures don't count
- Use `time.Now().UTC()` for all timestamps
- `ErrCircuitOpen` is a sentinel error for callers to detect circuit-open state
- Integration with SyncStatusTracker: add `CircuitState` to `ProviderSyncStatus` (deferred to 21.4)
