# Story 8.2: Obsidian Bidirectional Sync

Status: in-progress

## Story

**As an** Obsidian user,
**I want** changes made in Obsidian reflected in ThreeDoors and vice versa,
**so that** my tasks stay in sync regardless of where I edit them.

## Acceptance Criteria

1. **File Watcher Integration** (AC:1)
   - Given a configured Obsidian vault
   - When files are modified externally (e.g., user edits in Obsidian)
   - Then the file watcher detects changes via fsnotify

2. **Incremental Re-parse** (AC:2)
   - Given a file change is detected
   - When the watcher triggers a sync
   - Then only the changed file is re-parsed (not full vault reload)
   - And the in-memory task pool is updated with the changes

3. **Atomic Write Operations** (AC:3)
   - Given ThreeDoors writes a task change
   - When the file is written
   - Then atomic operations (write-to-tmp, sync, rename) prevent mid-write corruption
   - (Already implemented in 8.1 — verify still works with watcher)

4. **Concurrent Edit Handling** (AC:4)
   - Given both Obsidian and ThreeDoors modify the same file simultaneously
   - When a conflict is detected
   - Then last-write-wins semantics apply
   - And the conflict is logged for user awareness

5. **Sync Latency** (AC:5)
   - Given a file is modified externally
   - When the watcher is running
   - Then the change is detected and task pool updated within 2 seconds

6. **Watcher Lifecycle** (AC:6)
   - Given the adapter is configured with watching enabled
   - When the adapter starts
   - Then the watcher starts monitoring the vault directory
   - And when the adapter stops, the watcher is cleanly shut down

## Technical Context

### Prerequisites
- Story 8.1 complete (ObsidianAdapter with read/write capability)
- fsnotify library for cross-platform file watching

### Files to Create
- `internal/tasks/obsidian_watcher.go` — File watcher for Obsidian vault
- `internal/tasks/obsidian_watcher_test.go` — Tests for watcher

### Files to Modify
- `internal/tasks/obsidian_adapter.go` — Add watcher integration, callback support
- `go.mod` / `go.sum` — Add fsnotify dependency

### Design Decisions
- Use fsnotify for cross-platform file system notifications
- Debounce rapid events (e.g., 100ms) to handle editors that write in stages
- Self-write detection: ignore fsnotify events triggered by our own atomic writes
- Use a callback pattern for notifying the task pool of changes
- Watcher runs in a goroutine, communicates via channels
