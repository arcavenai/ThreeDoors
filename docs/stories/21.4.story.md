# Story 21.4: Sync Dashboard Enhancements

**Status:** draft
**Epic:** 21 — Sync Protocol Hardening

## User Story

As a ThreeDoors user,
I want to see per-provider sync health, staleness, and pending queue status,
So that I know which data is current and which may be stale.

## Acceptance Criteria

**Given** multiple providers are active with varying health states
**When** the sync status is displayed in the TUI
**Then:**

- AC1: `ProviderSyncStatus` gains fields: CircuitState, RetryIn, StaleSince, SyncCount24h, ErrorCount24h
- AC2: TUI sync status line shows circuit state icons: ✓ (closed/healthy), ✗ (open/error), ↻ (half-open/probing)
- AC3: Staleness indicator: providers exceeding staleness threshold show time since last sync
- AC4: WAL pending count displayed when items are queued: `⏳ WAL pending (N items, oldest Xm)`
- AC5: Per-provider display: `✓ textfile 5s ago | ✗ jira error (retry in 2m) | ↻ reminders probing...`
- AC6: Unit tests for status rendering at all circuit states and edge cases
- AC7: No new TUI views — extends existing `SyncStatusTracker` display logic

## Technical Notes

- Extend existing `ProviderSyncStatus` in `internal/core/sync_status.go`
- Staleness thresholds: 5 minutes for file-backed providers, 15 minutes for network providers
- Use Lipgloss styling for status icons (green ✓, red ✗, yellow ↻)
- WAL pending info: read from `WALProvider.PendingCount()` and `WALProvider.OldestPending()`
- SyncCount24h and ErrorCount24h: aggregate from SyncLog entries within last 24 hours
- Format time durations human-friendly: "5s", "2m", "1h 30m"
- All output through View() methods — no fmt.Println
