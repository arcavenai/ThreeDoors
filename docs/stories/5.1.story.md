# Story 5.1: macOS Distribution & Packaging

Status: Ready for Review

## Story

**As a** macOS user,
**I want** to install ThreeDoors via Homebrew, a signed/notarized binary, or a pkg installer,
**so that** I can install and run the application without Gatekeeper security warnings or manual build steps.

## Acceptance Criteria

1. **CI Code Signing with Apple Developer Certificate** (AC:1)
   - Given the CI pipeline produces macOS binaries (darwin/arm64, darwin/amd64)
   - When a release build is triggered on push to main
   - Then the macOS binaries are signed with a Developer ID Application certificate
   - And signing uses a macOS runner (`macos-latest`) since `codesign` requires macOS
   - And the certificate (.p12), password, Apple ID, app-specific password, and Team ID are stored as GitHub Actions encrypted secrets
   - And `codesign --verify --deep --strict` passes on signed binaries
   - And `--options runtime` (hardened runtime) is enabled, as required for notarization
   - And signing is skipped gracefully in forks or when secrets are unavailable (with informative log)
   - And unsigned linux/amd64 binary is unaffected
   - And a temporary keychain is created, used, and destroyed within the CI job

2. **Notarization Workflow for Apple Gatekeeper** (AC:2)
   - Given signed macOS binaries are produced by the CI pipeline
   - When the notarization step runs
   - Then the binaries are submitted to Apple's notarization service via `xcrun notarytool submit`
   - And the workflow waits for notarization completion with timeout (`--wait`)
   - And notarization failures produce clear CI error messages with the notarization log URL
   - And `spctl --assess --type execute` passes on notarized binaries
   - And notarization credentials (Apple ID, app-specific password, team ID) are stored as GitHub Actions secrets
   - Note: Standalone Mach-O binaries cannot be stapled; Gatekeeper checks notarization online

3. **Homebrew Tap Formula** (AC:3)
   - Given a new GitHub repository `arcaven/homebrew-tap` exists
   - When a user runs `brew install arcaven/tap/threedoors`
   - Then the correct signed+notarized binary is downloaded based on architecture (arm64 vs amd64)
   - And `threedoors` command is available on the user's PATH
   - And `brew upgrade arcaven/tap/threedoors` picks up new versions
   - And the formula includes SHA256 checksums for both architecture binaries
   - And the formula includes a `test` block that runs `threedoors --version` or `threedoors health`
   - And CI automatically updates the formula (SHA256 + version URL) on each release
   - And a `HOMEBREW_TAP_TOKEN` secret (GitHub PAT) enables pushing to the tap repo

4. **pkg Installer as Alternative Distribution** (AC:4)
   - Given signed and notarized binaries exist
   - When the release workflow runs
   - Then a `.pkg` installer is created using `pkgbuild` that installs `threedoors` to `/usr/local/bin/`
   - And the pkg identifier is `com.arcaven.threedoors`
   - And the pkg is signed with a Developer ID Installer certificate
   - And the pkg is notarized with Apple and the ticket is stapled (`xcrun stapler staple`)
   - And the pkg is uploaded as a GitHub Release asset alongside raw binaries
   - And double-clicking the pkg launches macOS Installer.app

### Local Development Without Apple Certs

For local testing of workflow logic without Apple Developer certificates:

- Set `SKIP_SIGNING=true` environment variable to skip signing/notarization steps

  ```makefile
  sign:
  ifndef APPLE_SIGNING_IDENTITY

  endif
  ```

## Dev Notes

### Version Strategy

**Format:** Semantic versioning (semver): `0.1.0`, `0.2.0`, `1.0.0`, etc.
**Build-time embedding:** Use Go linker flags: `-ldflags "-X main.version=0.1.0"`
**Output format:** `ThreeDoors 0.1.0` — the word "ThreeDoors" followed by space and version string.
**CI version:** For alpha releases from main, use format `0.1.0-alpha.YYYYMMDD.SHA7` (e.g., `0.1.0-alpha.20260302.d5e99a1`).
**Git tag:** Release tags follow format `v0.1.0` (with `v` prefix).
**Makefile default:** `VERSION ?= dev` — local builds show `ThreeDoors dev`.

### Key Technical Details

**Code Signing on macOS CI:**
- GitHub Actions `macos-latest` runners have Xcode and codesign pre-installed
- Certificates must be base64-encoded and stored as GitHub Actions secrets
- A temporary keychain must be created, used, and destroyed within the CI job
- `--options runtime` enables hardened runtime, required for notarization
- Signing identity format: `"Developer ID Application: Name (TEAMID)"`

**Notarization:**
- `xcrun notarytool` is the current CLI (replaces deprecated `altool`)
- App-specific passwords required (not regular Apple ID passwords)
- Notarization can take 5-15 minutes; use `--wait` with timeout
- Standalone Mach-O binaries **cannot** be stapled — Gatekeeper checks notarization online
- pkg files **can** be stapled after notarization

**Homebrew Formula:**
- Taps live in repos named `homebrew-<tapname>` (e.g., `homebrew-tap`)
- Formula files go in `Formula/` directory
- Binary formulas use `url` pointing to GitHub Release asset URLs
- SHA256 verification is mandatory
- Architecture detection: `Hardware::CPU.arm?` / `on_arm` / `on_intel` blocks

**pkg Creation:**
- `pkgbuild` creates component packages
- `productsign` signs with "Developer ID Installer" certificate (different from Application cert)
- pkg installer certificate is separate from codesign identity

### Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Apple Developer Program not enrolled | Must be done before implementation; $99/year |
| Certificate expiry (annual) | Document renewal process; CI fails loudly if expired |
| Notarization service downtime | Add retry logic; notarization is async |
| macOS runner costs (GitHub Actions) | macOS runners use more minutes; signing step is fast (~2 min) |
| Standalone binaries can't be stapled | Gatekeeper checks notarization online; works without stapling |

### Out of Scope

- Windows code signing (Authenticode)
- Linux packages (.deb, .rpm, snap, flatpak)
- Auto-update mechanism within the binary
- Universal binary (fat binary) — keep separate for smaller downloads
- DMG installer — pkg is preferred for CLI tools; DMG is for GUI apps
- Homebrew Cask — ThreeDoors is CLI, formula is correct

#### Go Wrapper Package: `internal/dist/`

To enable unit testing of CLI tool invocations, create a small Go package:

```
internal/dist/
├── dist.go           # Package-level types and interfaces
├── codesign.go       # Wraps codesign CLI calls
├── notarize.go       # Wraps xcrun notarytool calls
├── pkg_builder.go    # Wraps pkgbuild/productsign calls
├── codesign_test.go  # Tests with stubbed exec
```

**Interface for testability:**
```go
// CommandRunner allows stubbing exec.Command in tests
type CommandRunner interface {
    Run(name string, args ...string) ([]byte, error)
}

// RealRunner executes actual commands
type RealRunner struct{}

func (r RealRunner) Run(name string, args ...string) ([]byte, error) {
    return exec.Command(name, args...).CombinedOutput()
}

// StubRunner returns pre-configured responses for tests
type StubRunner struct {
    Responses map[string]struct {
        Output []byte
        Err    error
    }
}
```

This allows unit tests to verify correct command construction (flags, arguments, ordering) without requiring actual Apple Developer tools.
