# Story 24.2: Read-Only Task Resources & Query Tools

**Status:** draft
**Epic:** 24 — MCP/LLM Integration Server
**Depends on:** Story 24.1
**Blocks:** Stories 24.4, 24.6, 24.7

## User Story

As an LLM client connected via MCP,
I want to read task data, query tasks with filters, and inspect provider health,
So that I can understand the user's task landscape and answer questions about their work.

## Acceptance Criteria

**Given** the MCP server is running and connected to ThreeDoors core
**When** an LLM client requests task data
**Then:**

- AC1: MCP Resources registered: `threedoors://tasks` (all tasks), `threedoors://tasks/{id}` (single task), `threedoors://tasks/status/{status}` (by status), `threedoors://tasks/provider/{name}` (by provider)
- AC2: MCP Resources registered: `threedoors://providers` (registry + health), `threedoors://providers/{name}/status` (individual health)
- AC3: MCP Resources registered: `threedoors://session/current` (current session metrics), `threedoors://session/history` (historical sessions)
- AC4: MCP Tool `query_tasks` implemented with filter args: `status`, `type`, `effort`, `provider`, `text_contains`, `created_after`, `created_before`, `limit`, `sort_by`, `sort_order`
- AC5: MCP Tool `get_task` implemented — returns full task detail including enrichment data for a given task ID
- AC6: MCP Tool `list_providers` implemented — returns configured providers with health status and sync freshness
- AC7: MCP Tool `get_session` implemented — returns current or historical session metrics
- AC8: All query responses include metadata: `total_count`, `returned_count`, `query_time_ms`, `providers_queried`, `data_freshness`
- AC9: `TaskQueryEngine` with `Search(query string, opts SearchOptions) ([]SearchResult, error)` for text-based queries with token overlap scoring, field weighting (text 3x, context 2x, notes 1x), and recency boost
- AC10: Unit tests for all resources, tools, query engine, and response metadata

## Technical Notes

- Resources are strictly read-only — they call existing `TaskPool`, `Registry`, and `SessionTracker` methods
- `TaskQueryEngine` lives in `internal/mcp/query.go` — wraps `TaskPool.Filter()` with scoring and search
- `SearchResult` struct: `Task`, `Score` (0.0-1.0), `MatchedOn` (field names), `Highlights` (fragments)
- Response metadata enables LLMs to paginate and understand data freshness
- Field-weighted search uses Jaccard similarity on tokenized text
- No MCP subscriptions in this story — just request/response reads
