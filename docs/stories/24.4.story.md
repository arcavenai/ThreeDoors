# Story 24.4: Proposal Store & Controlled Enrichment API

**Status:** Done (PR #TBD)
**Epic:** 24 — MCP/LLM Integration Server
**Depends on:** Stories 24.2, 24.3
**Blocks:** Story 24.5

## User Story

As an LLM client,
I want to propose task enrichments (metadata, subtasks, context, relationships) through a controlled API,
So that I can suggest improvements without directly modifying task data.

## Acceptance Criteria

**Given** the core design principle that LLMs propose and users approve
**When** the proposal system is implemented
**Then:**

- AC1: `Proposal` struct defined with fields: `ID` (UUID), `Type` (ProposalType), `TaskID`, `BaseVersion` (time.Time — task's UpdatedAt when proposal created), `Payload` (json.RawMessage), `Status` (pending/approved/rejected/expired/stale), `Source` (MCP client identifier), `Rationale`, `CreatedAt`, `ReviewedAt`, `ExpiresAt` (7-day default)
- AC2: `ProposalType` constants: `enrich-metadata`, `add-subtasks`, `add-context`, `add-note`, `suggest-blocker`, `suggest-relationship`, `suggest-category`, `update-effort`
- AC3: `ProposalStore` backed by append-only JSONL (`~/.threedoors/proposals.jsonl`) with methods: `Create(proposal Proposal) error`, `Get(id string) (Proposal, error)`, `List(filter ProposalFilter) []Proposal`, `UpdateStatus(id string, status ProposalStatus, reviewedAt time.Time) error`
- AC4: Optimistic concurrency control: proposals include `BaseVersion`; on review, compare to task's current `UpdatedAt` — mark `stale` if they differ
- AC5: MCP Tools registered: `propose_enrichment(task_id, type, payload, rationale)`, `suggest_task(text, context, effort, type, rationale)`, `suggest_relationship(from_id, to_id, relation_type, rationale)`
- AC6: MCP Resource `threedoors://proposals/pending` returns all pending proposals
- AC7: Deduplication at intake: reject proposals matching existing tasks (text similarity), other pending proposals, recently rejected proposals (within 7 days), or completed tasks
- AC8: Per-task cap of 5 pending proposals enforced
- AC9: Proposal expiration: background sweep marks proposals older than 7 days as `expired`
- AC10: `IntakeChannel` interface defined: `Name() string`, `Suggest(ctx context.Context, proposal Proposal) error`, `HealthCheck() HealthCheckResult`
- AC11: Unit tests for proposal CRUD, optimistic concurrency, deduplication, expiration, and per-task caps

## Technical Notes

- Proposal store is append-only JSONL, consistent with session tracking patterns
- `Payload` is `json.RawMessage` to allow type-specific validation per `ProposalType`
- Stale proposals require re-evaluation by the user, not automatic rejection
- The `IntakeChannel` interface enables future intake sources (GitHub issues, calendar events, codebase analysis) — only implement `LLMIntakeChannel` in this story
- Deduplication leverages existing `DuplicateDetector` from core
- Source tracking format: `"mcp:claude-desktop"`, `"mcp:cursor"`, etc.
