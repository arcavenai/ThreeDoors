# Story 19.4: Jira Config and Registration

**Status:** Done (PR #153)
**Epic:** 19 — Jira Integration

## User Story

As a ThreeDoors user,
I want to configure Jira integration in my config.yaml,
So that I can connect to my Jira instance with my preferred JQL filter.

## Acceptance Criteria

**Given** the Jira adapter is implemented
**When** a user adds Jira config to config.yaml
**Then:**

- AC1: Config section: `url`, `auth_type` (basic|pat), `jql`, `max_results`, `poll_interval`
- AC2: Environment variable fallback: `JIRA_URL`, `JIRA_EMAIL`, `JIRA_API_TOKEN`
- AC3: Config validation: required fields (url, auth_type), URL format, auth_type enum
- AC4: Sensible defaults: jql = `"assignee = currentUser() AND statusCategory != Done"`, max_results = 50, poll_interval = 30s
- AC5: Adapter factory wired to config parsing in `internal/adapters/jira/config.go`
- AC6: Registration in `RegisterBuiltinAdapters()` as `"jira"`
- AC7: Unit tests for config parsing, validation, env var fallback, and default values

## Technical Notes

- Config follows existing pattern: `providers: [{name: jira, settings: {...}}]`
- Settings map keys: `url`, `auth_type`, `jql`, `max_results`, `poll_interval`, `email`, `api_token`
- Env vars take precedence over config.yaml settings
- Never store credentials in task YAML files (NFR22)
- Use `os.Getenv` for env var lookup — no third-party config library
- Validate URL is well-formed (must have scheme and host)
- auth_type must be "basic" or "pat"
