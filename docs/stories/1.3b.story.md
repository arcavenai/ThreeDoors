# Story 1.3b: Expand & Fork Actions in Detail View

Status: Done (PR #134)

## Story

As a user,
I want to expand a task into subtasks or fork (duplicate) a task from the detail view,
so that I can break down complex tasks or create variations without leaving the TUI.

## Background

Story 1.3 (Task 6) deferred the Expand and Fork actions. Currently, pressing 'E' or 'F' in the detail view shows a flash message: "Expand not yet implemented" / "Fork not yet implemented" (see `internal/tui/detail_view.go:122-127`).

## Acceptance Criteria

1. **Expand ('E' key):** When pressed in detail view, transitions to a text input mode. User types subtask text, presses Enter. A new task is created with that text and added to the in-memory pool and persisted via the provider.
2. **Expand cancellation:** Pressing Esc during expand input returns to the detail view without creating a task.
3. **Expand empty input:** Pressing Enter with empty text shows a flash message ("Subtask text cannot be empty") and stays in expand input mode.
4. **Fork ('F' key):** When pressed in detail view, immediately duplicates the current task's text as a new task, adds it to the pool, persists via provider, and shows a flash confirmation.
5. **Fork creates independent copy:** The forked task gets a new UUID, fresh timestamps, and "todo" status regardless of the original task's status.
6. **Post-action behavior:** After successful expand or fork, return to the doors view with refreshed doors.
7. **Error handling:** If persistence fails, show a flash error but don't crash. The task should still be in the in-memory pool.

## Dev Notes

### Architecture Decisions

- **Expand input reuses the blocker input pattern**: Simple inline text capture within DetailView (same approach as `DetailModeBlockerInput`), not a separate view.
- **Fork uses existing `TaskAddedMsg`**: No new message type needed for fork — the main_model.go already handles `TaskAddedMsg` by adding to pool and persisting.
- **Expand needs `ExpandTaskMsg`**: Unlike fork, expand creates a task from user input within the detail view, so we need a distinct message that the main model handles to create the task, add to pool, and persist.
