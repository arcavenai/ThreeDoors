# Story 23.5: Health, Version Commands and Exit Code Enforcement

**Status:** Done
**Epic:** 23 — CLI Interface
**Depends on:** Story 23.1

## User Story

As a CLI user or LLM agent,
I want `threedoors health` and `threedoors version` commands with structured output and consistent exit codes,
So that I can verify system status and integrate ThreeDoors into health-check scripts.

## Acceptance Criteria

**Given** the CLI scaffold from Story 23.1 is in place
**When** health and version commands are implemented
**Then:**

- AC1: `threedoors health` runs all health checks via `HealthChecker.RunAll()` and displays results as a table: Check Name, Status (OK/WARN/FAIL), Message
- AC2: `threedoors health --json` outputs JSON envelope with `data.overall`, `data.duration_ms`, `data.checks[]`
- AC3: `threedoors health` exits 0 if all checks pass, exit code 4 if any check fails
- AC4: `threedoors version` displays version, commit hash, build date, and Go version in human-readable format
- AC5: `threedoors version --json` outputs version info as JSON envelope
- AC6: Version info populated via `ldflags` at build time (add to Makefile)
- AC7: All CLI commands consistently use the exit code scheme (0-5) defined in Story 23.1
- AC8: Unit tests verify exit codes for success, not-found, validation error, provider error, and ambiguous input scenarios

## Technical Notes

- `HealthChecker` is in `internal/core/health_checker.go` — already CLI-ready with JSON-serializable results
- Version variables should be package-level `var` in `internal/cli/version.go`, set via `-ldflags` in the Makefile
- Makefile should add `-ldflags "-X internal/cli.Version=... -X internal/cli.Commit=... -X internal/cli.BuildDate=..."`
- Reference: `docs/research/cli-interface-design.md` sections 4, 7
