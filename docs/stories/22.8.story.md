# Story 22.8: Safety Guardrails (Rate Limiting, Cost Caps, Audit Log)

**Status:** Done (PR #160)
**Epic:** 22 — Self-Driving Development Pipeline
**Depends on:** Story 22.2

## User Story

As a user,
I want safety guardrails preventing runaway agent spawning and providing an audit trail,
So that I can use the self-driving pipeline without risk of excessive cost or uncontrolled automation.

## Acceptance Criteria

**Given** the dispatch engine is operational
**When** guardrails are configured
**Then:**

- AC1: Max concurrent workers enforced (default 2) — before each dispatch, `Dispatcher.ListWorkers` is called; if count >= `max_concurrent`, dispatch is refused with message "Max concurrent workers reached (N/N). Wait for a worker to complete."
- AC2: Manual approval gate by default (`auto_dispatch: false`) — pending items require explicit 'y' approval in the dev queue view before worker creation
- AC3: Minimum 5-minute cooldown between dispatches to the same task — tracked by `QueuedAt` timestamp on the task's most recent queue item
- AC4: Daily dispatch limit enforced (default 10) — counts dispatches in the current UTC day from the audit log; dispatch refused with message "Daily dispatch limit reached (N/N)" if exceeded
- AC5: Every dispatch, completion, and failure logged to `~/.threedoors/dev-dispatch.log` in JSONL format with fields: timestamp, event_type (dispatch/complete/fail/kill), task_id, queue_item_id, worker_name, pr_number, error
- AC6: `:dispatch --dry-run` flag shows the full multiclaude command that would be executed, without executing it — displayed in the TUI as a preview
- AC7: Guardrail settings configurable in `~/.threedoors/config.yaml` under `dev_dispatch` section: `max_concurrent` (int), `auto_dispatch` (bool), `cooldown_minutes` (int), `daily_limit` (int)
- AC8: All guardrail violations produce user-visible messages in the TUI — displayed as styled status messages, not silent failures

## Technical Notes

- Guardrail checks are called before each dispatch — they are a validation layer between the approval action and the actual `CreateWorker` call
- Audit log format follows the existing JSONL pattern used by `sessions.jsonl`
- Audit log file uses append mode — no atomic write needed (append is atomic for small writes on POSIX)
- Config structure:
  ```yaml
  dev_dispatch:
    enabled: false
    max_concurrent: 2
    auto_dispatch: false
    cooldown_minutes: 5
    daily_limit: 10
    require_story: false
  ```
- Dry run mode shows the command in a TUI overlay/message — not stdout
- Daily limit resets at UTC midnight — count dispatches where `timestamp >= today_start_utc`
- Cooldown is per-task, not global — the same task cannot be re-dispatched within 5 minutes, but different tasks can be dispatched immediately
