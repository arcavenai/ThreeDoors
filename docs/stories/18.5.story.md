# Story 18.5: CI Integration for Docker E2E Tests

## Status: done

## Story

As the team,
I want Docker E2E tests running automatically in CI,
So that TUI regressions are caught on every PR without relying on manual testing.

## Technical Notes

- **GitHub Actions Docker support:** Use `docker compose run -T` (no TTY in CI) in a workflow step
- **Docker layer caching:** Use `docker/setup-buildx-action` + `docker/build-push-action` with `cache-from`/`cache-to` using GitHub Actions cache backend, or simpler: use `actions/cache` with Docker buildx bake
- **Parallel execution:** Define `test-docker-e2e` as a separate job with no `needs` dependency on `quality-gate` тАФ GitHub Actions runs independent jobs in parallel by default
- **Artifact upload:** Use `actions/upload-artifact@v4` to upload `test-results/` directory
- **Golden file diffs:** After test run, check for modified golden files and output diff if any exist
- **Exit code propagation:** `docker compose run` propagates the container exit code by default; `--rm` cleans up after
- **No Go code changes needed** тАФ this story only modifies `.github/workflows/ci.yml`
- **Existing workflow structure:** `quality-gate` тЖТ `build-binaries` тЖТ `sign-and-notarize` тЖТ `release` тЖТ `update-homebrew`. The new job should be independent of this chain.

## Dev Notes

- The `build-binaries` job has `needs: quality-gate` тАФ but `test-docker-e2e` should NOT depend on it
- `docker compose run -T` disables TTY allocation (CI has no terminal)
- teatest creates its own pseudo-TTY via `teatest.WithInitialTermSize()` тАФ works without Docker TTY
- The `--rm` flag on `docker compose run` removes the container after execution
- Coverage output goes to `test-results/coverage.out` per Story 18.4's docker-compose.test.yml
