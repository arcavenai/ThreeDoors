# Story 1.3a: Quick Search & Command Palette

Status: ready-for-dev

## Story

As a user,
I want to search for specific tasks and execute commands via text input,
so that I can efficiently find and act on tasks without relying solely on the three doors.

## Acceptance Criteria

1. **Search Activation (/ key):** Given the user is in the three doors view, when the user presses '/', then a text input field appears at the bottom with placeholder "Search tasks... (or :command for commands)"
2. **Live Search Results:** Given the search input is active, when the user types characters, then matching tasks display from bottom-up (live substring matching), the list updates with each keystroke, if no matches "No tasks match '<text>'" is shown, and empty input shows no results
3. **Search Result Navigation:** Given search results are displayed, when the user navigates with Arrow Up/Down keys, then the selected result is highlighted (wrapping at boundaries), and pressing Enter opens the selected task in the detail view (reusing Story 1.3's DetailView). Note: WASD/HJKL navigation is deferred because those keys conflict with text input in the search field — only arrow keys navigate results.
4. **Context-Aware Return from Detail:** Given a task detail was opened from search, when the user presses Esc from the detail view, then the search view returns with search text preserved and previous selection highlighted
5. **Search Dismissal:** Given the search input is active (not in detail view), when the user presses Esc, then search mode closes and returns to three doors view. Note: Ctrl+C always quits the application (global handler in MainModel), it does NOT close search — this is consistent with existing behavior.
6. **Command Mode Trigger:** Given the search input is empty, when the user types ':' as the first character, then the input switches to command mode with visual indicator
7. **:add Command:** Given command mode is active, when the user types `:add <task text>` and presses Enter, then a new task is added to tasks.yaml via the existing persistence layer, flash shows "Task added", and the user remains in search/command mode (query is cleared, ready for next command). If text is empty (`:add` or `:add `), show flash "Usage: :add <task text>"
8. **:mood Command:** Given command mode is active, when the user types `:mood [mood]` and presses Enter, then: if a valid mood arg is provided (one of: Focused, Tired, Stressed, Energized, Distracted, Calm), it is logged via `tracker.RecordMood(mood, "")` with empty customText, then flash shows "Mood logged: <mood>" and search/command mode is cleared. If no arg given (`:mood`), the mood dialog opens via `ShowMoodMsg{}`. If an invalid mood arg is given, show flash "Invalid mood. Valid: Focused, Tired, Stressed, Energized, Distracted, Calm"
9. **:stats Command:** Given command mode is active, when the user types `:stats` and presses Enter, then session statistics are displayed as a flash message with format: "Tasks completed: X | Doors viewed: Y | Refreshes: Z | Session: Nm" (duration in minutes, computed as `time.Since(metrics.StartTime)`)
10. **:help Command:** Given command mode is active, when the user types `:help` and presses Enter, then available commands and key bindings are displayed as a flash message with this exact text:
    ```
    Commands: :add <text> | :mood [mood] | :stats | :help | :quit
    Keys: a/←/w/↑/d/→ select door | s/↓ re-roll | Enter open | / search | M mood | q quit
    In search: ↑/↓ navigate | Enter select | Esc back
    ```
11. **:quit Command:** Given command mode is active, when the user types `:quit` or `:exit` and presses Enter, then the application exits cleanly
12. **Unknown Command:** Given command mode is active, when the user types an invalid command and presses Enter, then "Unknown command: '<command>'. Type :help for available commands." is shown

## Dev Notes

### Architecture Patterns & Constraints

- **Two-layer architecture**: TUI layer (`internal/tui/`) imports domain layer (`internal/tasks/`). NEVER the reverse.
- **Bubbletea pattern**: The existing codebase uses pointer receivers and a delegation pattern (NOT the classic Elm-architecture value receivers). `MainModel` is `*MainModel`, sub-views are `*DoorsView`, `*DetailView`, `*MoodView`. Sub-views have `Update(msg) tea.Cmd` (returning only a Cmd, not tea.Model). Follow this EXACT pattern.
- **Message-driven communication**: Views communicate via message types defined in `messages.go`. Sub-views return `tea.Cmd` functions that produce messages. `MainModel.Update()` handles these messages at the top level.
- **Constructor pattern**: `NewXxxView(dependencies)` for production. Tests use the constructors directly with test data.

### Key Design Decisions

1. **SearchView as a new sub-view**: Follows the same pattern as `DoorsView`, `DetailView`, `MoodView`. Pointer receiver with `Update(msg) tea.Cmd` and `View() string`.

2. **No bubbles/textinput dependency**: The existing codebase handles text input manually (see `DetailView.handleBlockerInput()` and `MoodView.handleCustomInput()`). Follow the SAME pattern for search input — manual character handling with backspace support. Do NOT add bubbles as a dependency.

3. **ViewMode Extension**:
```go
const (
    ViewDoors  ViewMode = iota
    ViewDetail
    ViewMood
    ViewSearch  // NEW
)
```

4. **SearchView Struct**:
```go
type SearchView struct {
    query           string
    results         []*tasks.Task
    selectedIdx     int        // -1 = no selection, 0+ = result index
    pool            *tasks.TaskPool
    tracker         *tasks.SessionTracker
    isCommandMode   bool       // true when query starts with ':'
    width           int
}
```

5. **Context-Aware Esc Navigation**: MainModel needs a `fromSearch bool` field. When opening detail from search, set `fromSearch = true`. When Esc is pressed in detail view:
   - If `fromSearch == true`: return to search view (restore state)
   - If `fromSearch == false`: return to doors view (existing behavior)

6. **Message Flow for Search → Detail → Search**:
```
User presses '/' in doors → MainModel creates SearchView, sets ViewSearch
User types query → SearchView filters tasks live
User presses Enter on result → SearchView returns OpenDetailFromSearchMsg
MainModel receives OpenDetailFromSearchMsg → creates DetailView, sets ViewDetail, fromSearch=true
User presses Esc in detail → DetailView returns ReturnToDoorsMsg
MainModel intercepts ReturnToDoorsMsg when fromSearch=true → sets ViewSearch, calls searchView.updateResults(), fromSearch=false
```

**CRITICAL: View Transition Edge Cases:**
- **Mood from search-detail:** If user opens detail from search, then presses M → mood view opens. On mood capture, MainModel should check if `searchView != nil` and return to ViewSearch (not ViewDoors).
- **Task completed from search-detail:** If user completes a task from search-initiated detail, `TaskCompletedMsg` is handled normally (task removed, doors refresh). The search view is cleared and user returns to doors — do NOT return to search because the task pool changed.
- **Stale results on return:** When returning from detail to search, ALWAYS call `updateResults()` to re-filter against current pool state. Clamp `selectedIdx` to `min(savedIdx, len(results)-1)` to avoid out-of-bounds.

7. **:add Command Integration with Persistence**:
```go
// In SearchView's command handler:
case "add":
    if args == "" {
        return func() tea.Msg { return FlashMsg{Text: "Usage: :add <task text>"} }
    }
    newTask := tasks.NewTask(args)
    sv.pool.AddTask(newTask)
    // Persistence: signal MainModel to save
    return func() tea.Msg {
        return TaskAddedMsg{Task: newTask}
    }
```
Add `TaskAddedMsg` to messages.go. MainModel handles it by calling `saveTasks()` and showing flash.

8. **Bottom-Up Result Display**: Results are rendered with the last match at the bottom (near the input field), first match at the top. This reduces eye travel per UX design goals.

9. **Live Substring Matching**: Case-insensitive using `strings.Contains(strings.ToLower(task.Text), strings.ToLower(query))`. Filter runs on every keystroke against `pool.GetAllTasks()`. Cap results at `maxResults = 20`. If more matches exist, append "... and N more results" indicator. With typical task pools (<1000), this completes in microseconds.

10. **Stats Display**: For simplicity, `:stats` shows a flash message with key metrics. A dedicated stats overlay view is out of scope — use `FlashMsg` with a longer duration or a multi-line flash.

### Navigation Key Handling in Search Results

**Arrow keys ONLY** for result navigation (WASD/HJKL deferred — they conflict with text input):
- **Arrow Up**: Move selection up (decrease index; at 0, wrap to last result)
- **Arrow Down**: Move selection down (increase index; past last, wrap to 0)
- **Enter**: Open selected result in detail view (no-op if `selectedIdx == -1`)
- **Any printable character**: Goes into query, resets `selectedIdx = -1`, triggers `updateResults()`

This is consistent with AC 3 which specifies Arrow Up/Down navigation only.

```go
    parts := strings.SplitN(strings.TrimSpace(cmdStr), " ", 2)
    cmd := strings.ToLower(parts[0])
    args := ""

    }

    switch cmd {
    case "add":
        // ...
    case "mood":
        // ...
    case "stats":
        // ...
    case "help":
        // ...
    case "quit", "exit":
        return tea.Quit
    default:
        return func() tea.Msg {

        }
    }
}
```

### Search Result Rendering (Bottom-Up)

```go
func (sv *SearchView) View() string {
    s := strings.Builder{}

    // Results rendered from top to bottom, but the visual effect is bottom-up
    // because the search bar is at the bottom
    if sv.query != "" && len(sv.results) == 0 {
        s.WriteString(noResultsStyle.Render(fmt.Sprintf("No tasks match '%s'", sv.query)))
        s.WriteString("\n")
    }

    for i, task := range sv.results {
        prefix := "  "
        style := searchResultStyle
        if i == sv.selectedIdx {
            prefix = "> "
            style = searchSelectedStyle
        }
        s.WriteString(style.Render(fmt.Sprintf("%s%s [%s]", prefix, task.Text, task.Status)))
        s.WriteString("\n")
    }

    // Search input at bottom
    prompt := "Search: "
    if sv.isCommandMode {
        prompt = commandModeStyle.Render("CMD: ")
    }
    s.WriteString("\n")
    s.WriteString(prompt + sv.query + "_")

    return s.String()
}
```

```go
assertFlashMsg(t, cmd, "Tasks completed:")  // not exact match
```

**SearchView Method Signatures (for TEA reference)**:

```go
func NewSearchView(pool *tasks.TaskPool, tracker *tasks.SessionTracker) *SearchView
func (sv *SearchView) SetWidth(w int)
func (sv *SearchView) Update(msg tea.Msg) tea.Cmd
func (sv *SearchView) View() string
func (sv *SearchView) updateResults()        // private — called internally after query changes
func (sv *SearchView) executeCommand() tea.Cmd // private — called on Enter in command mode
func (sv *SearchView) RestoreFromDetail()     // public — called by MainModel to restore state after detail view
```

**Nil Tracker Handling**: `tracker` can be nil (consistent with DoorsView and DetailView patterns). All tracker calls must be guarded: `if sv.tracker != nil { sv.tracker.RecordMood(...) }`. Test with nil tracker to verify no panics.

**`updateResults()` is called inside `Update()`**: After every character input or backspace, `Update()` calls `updateResults()` internally. Tests verify results by checking `sv.results` or `sv.View()` after sending key messages. TEA does NOT need to call `updateResults()` separately in tests.

### Error Handling

- `:add` with empty text → show flash "Usage: :add <task text>"
- `:add` save failure → show flash "Error saving task: <error>" (do not crash)
- Search on empty pool → show "No tasks match" for any query
- Ctrl+C from any search state → quit cleanly

### Performance

- Live filtering on each keystroke is O(n) over all tasks. With typical task pools (<1000), this completes in microseconds.
- No debouncing needed for search input.
